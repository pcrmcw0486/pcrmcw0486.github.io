### Redis와 정합성관리 개요
레디스는 보조도구로서 활용되지, SSOT(Single source of Truth)이 아님을 인지해야한다.
그리고 아니어야한다고 생각한고 설계해야한다.
즉, Redis는 "언제든 데이터가 사라질 수 있고, 완벽한 정합성을 보장하기 어렵다"는 가정하에 설계해야함.
- 보조도구로써의 역할,Cache & Seesion store & Rate limiter & Lock & Temporary data
* 해당글에서는 redis를 예시로 들고 있지만.. 유사한 컨셉이라면 생길 수 있는 문제라고 보는게 맞다.

이에, DB 또는 특정한 원천 소스로 부터의 정합성 문제가 생길 수 있다.
ex) DB는 업데이트 되었지만, redis는 업데이트가 되지 않은 상황 등.
 DB = 100 & redis = 90
이를 어떻게 맞출거냐? 라는 질문이 "Redis의 정합성 관리" 라고 볼 수 있다.

### Cache stale
 - DB update 되었지만, redis는 업데이트 되지 않음.
 * stale이란 오래된 데이터를 의미한다.
 * 위의 예시에서 redis =90 으로 남아있는 것.

### (단순하게)해결방안
1. write-through or write-back
- write-through : write시 redis -> db 또는 db-> redis 순서로 모두 쓴다.
    - 장점: 항상 보장됨 (근데 이것도 무조건 보장된다고 볼 수있는가? DB update중 문제면..?)
    - 단점: latency 증가.
- write-back : redis에 먼저 써두고, DB에 비동기로 반영한다. 
    -단점: redis 장애 시, 데이터 유실 가능성.
2. Cache Aside + TTL 
데이터를 읽을 때, redis(missing) -> DB -> redis set
TTL을 설정하므로써, 자동으로 만료되어 자연스럽게 최신데이터 갱신을 유도. 
또는 write시 cache invalidate (무효화)하여 업데이트에도 최신 데이터 갱신을 유도
- TTL만 설정 시, TTL간 stale값 반환
3. 이벤트기반 캐시 무효화
- 특정 이벤트가 발생했을 때, 이벤트를 구독한 서버에서 캐시를 무효화하게 되면, 다시 최신화된 값을 로드하기를 기대한다.

race condition이 발생하면요?

### 만약 더 빠른 속도를 위해서 Redis 캐시만 사용하고 싶지 않다면요..?
분산환경에서 Redis라는 공통의 캐시를 사용할 수도 있겠지만, L1/L2 캐시처럼 LocalCache와 연계하여 사용할 수도 있고, 
굳이 Redis까지 안쓰더라도, 각 Pod들에서 Local cache만 사용할 수도 있다. 이럴때는 어떻게 캐시의 정합성을 맞출 수 있을까?

*즉, Redis 하나만 바꾼다고 다 해결되지 않는 케이스에 대해서도 고민해볼 수 있다.*
레디스와 정합성을 넘어서서 pod간 동기화의 문제로 볼수도 있기에 여기서는 작성하지 않는다.

### 실무 경험
예를들어 잘 변하지 않는 설정값들이 있다고 가정하자. 특정 기능의 Provider 업체가 두 곳이 있고, 이를 가중치 기반으로 라우팅해서 사용하다가, 특정 업체 Provider A가 특정기간동안 서비스 중단을 알려왔다. 이 때, 어드민 또는 백도어 등을 통해 Provider A의 가중치를 0으로 만들었는데..
* 이벤트 트리거는 별개의 문제긴하지만..

기능상 각 Provider의 가중치만 알면되고, 특정 동작에 자주 사용된다. + 변경사항이 매우 적음. 
이라 성능에 조금 더 초점을두었기에 redis를 고려하지 않고, 분산환경에서 로컬 캐시 (Caffeine)을 이용했다. (웜업시 세팅해도 되고..)
이렇게 되면 캐시 데이터 정합성 문제가 pod간 동기화 문제로 연계된다.

#### 1. Stale 5분 허용
처음에는 DB변경과 무관하게 TTL 5분을 두고 개발했었고,  5분이 지나면 항상 최신의 값을 불러올 수 있도록 했다. 
단점: 즉시 반영이 아니기에 TTL간 장애가 발생할 수 있다. + 관련 기능을 개발할때 DB 설정값을 변경하며 테스트하는 경우, 항상 5분을 기다리거나 서버를 다시 기동해야한다.

#### 2. 이벤트 기반 pod간 동기화 (캐시 무효화 이벤트 발행)
어드민을 통한 값 수정 또는 특정 이벤트 트리거를 통해 캐시 무효화 이벤트를 발행하여, consume하는 즉시 무효화하게 한다.
- 여러 이벤트 기반 도구들을 사용할 수 있으나, redis가 이미 연결되어 있었기에 redis pub/sub & redis streams를 고려했다.
- 보통은 kafka가 붙어있을 수 있어서 kafka를 이용하는 것도 하나의 방법일 수 있지만, 현재 구조에서 큰 리소스 없이 가볍고 빠르게 사용가능하다고 생각했다. 
- 이 때, stale의 허용 정도에 따라서 redis pub/sub은 고려해보아야한다.
실제로 redis pub/sub을 이용한 keyspace 기반 만료이벤트 발행 시에도, 만료 시점에 바로 발행되지 않을 뿐더러, redis pub/sub은 fire-and-forgot 이기에 모든 pod가 정상적으로 반영되었는지 알기 어렵다. 
다만, 빈도가 잦고, 알림과 같이 어느정도 허용가능한 범위라면 가볍게 사용할수는 있겠다.


### 마치면서..
여기서 생각할건 cache는 애초에 strong consistent를 목적으로 사용하지 않고, eventual consistency를 중요하게 생각한다. 즉 stale이 존재할 수 밖에 없다.

추가로.. 패턴이라고 하면 거창하고, 사실 생각하면 다 고민하고 나왔을 수 있는 해결방안들이지만 개발 간 소통을 위해서 거창하더라도 명칭을 알면 좋을것 같다.
